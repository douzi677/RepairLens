# RepairLens 产品需求文档 (PRD)

> **Know before you pay — 付款之前，先知道报价是否合理。**
>
> 版本：MVP v1.0 | 作者：AI PM | 日期：2026-07-08

---

## 1. 产品概述

RepairLens 是一款面向普通消费者的 AI 家电维修报价分析助手。用户只需选择家电品类和维修项目，输入维修师傅给出的报价，系统即可基于结构化价格数据与 AI 解释能力，生成一份清晰、可信的报价分析报告。

**RepairLens 是什么：** 维修决策支持工具（Decision Support Tool）

**RepairLens 不是什么：** AI 聊天机器人、家电维修百科、官方报价平台

---

## 2. 产品背景与用户痛点

### 市场背景

家电维修行业长期存在严重的信息不对称。普通消费者对零件成本、人工费用缺乏认知，面对维修师傅的报价往往只能"凭感觉"判断是否合理。市面工具要么是纯内容社区（小红书搜索经验帖），要么是纯交易平台（58同城），缺乏一个**轻量、即时、有数据支撑的报价分析工具**。

### 核心用户痛点

| 痛点 | 描述 |
|------|------|
| **信息黑盒** | 用户完全不知道一个"压缩机更换"应该花多少钱 |
| **决策焦虑** | 维修师傅站在门口等答复，用户被迫在几十秒内做几百元的决定 |
| **信任缺失** | 用户天然怀疑维修报价，但缺乏客观依据来验证 |
| **比价成本高** | 打电话问品牌售后、问多家维修店，沟通成本极高 |

### 一句话价值主张

> **用 30 秒输入，换来一次有数据支撑的维修决策信心。**

## 3. 用户画像

### 主要用户：城市租房/持家青年（25-40 岁）

- 非技术背景，对家电内部结构不了解
- 手机重度用户，期望移动端快速操作
- 价格敏感但不抠门——愿意付合理价格，反感被"宰"
- 使用场景：维修师傅上门或电话报价后，快速掏出手机判断是否合理

### 次要用户：中老年家庭用户（40-60 岁）

- 子女可能代为操作
- 更需要通俗易懂的结论和明确的下一步建议
- 对"官方建议"有天然信任感

## 4. 用户需求分析

### 用户故事

- 作为用户，我希望能**快速选择家电类型和故障**，不用打很多字
- 作为用户，我希望能**看到零件大概多少钱、人工大概多少钱**，而不是一个笼统的数字
- 作为用户，我希望能**知道师傅报的价格是高是低**，以及高出多少
- 作为用户，我希望能**得到下一步该怎么做**的具体建议，而不是一个冷冰冰的结论

### 非目标（本产品故意不做的事）

- 不做维修师傅推荐/派单
- 不做家电故障诊断（这是维修师傅的工作）
- 不做"全网最低价"承诺
- 不替代专业维修判断

## 5. 产品目标

### 北极星指标

**报告有用率** — 用户查看报告后点击 👍"有帮助"的比例（👍 / (👍 + 👎)）。该指标直接衡量产品是否真正帮助用户做出了更好的维修决策。

> 在每份分析报告底部嵌入 👍/👎 反馈按钮，作为北极星指标的数据采集点。

### MVP 成功标准

- 用户完成一次分析的平均时间 < 60 秒
- 报告生成时间 < 3 秒
- 报告有用率（👍 比例）≥ 70%

## 6. MVP 范围

### MVP 包含

- 6 大家电品类（空调、冰箱、洗衣机、电视、热水器、其他）
- 每品类预设 5-10 个高频维修项目
- 自定义维修项目输入（支持自由文本）
- 设备型号输入（可选跳过）
- 基于本地 JSON 数据的结构化价格计算
- AI 生成的个性化分析解释和建议
- 移动端优先的单页 Wizard 交互

### MVP 不包含

- 电商价格实时同步
- OCR 维修单识别
- 用户账号系统
- 历史记录

## 7. 用户流程 (User Flow)

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  选择家电品类  │ → │  (可选)输入型号 │ → │  选择维修项目  │
│  6个品类卡片  │    │  可跳过       │    │  多选+自定义   │
└──────────────┘    └──────────────┘    └──────────────┘
                                                ↓
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  查看分析报告  │ ← │  AI分析+计算  │ ← │ 输入维修报价   │
│  完整报告+建议 │    │  服务端处理   │    │  师傅给的报价  │
└──────────────┘    └──────────────┘    └──────────────┘
```

**流程设计原则：** 每步只做一件事，5 步完成，支持任意步骤返回修改，全程无需注册登录。

### "其他"品类的特殊处理逻辑

当用户选择"其他"作为家电品类时：

1. **不展示预设维修项目** — "其他"品类在价格数据库中无预设维修项（`repair_items` 为空数组）
2. **用户必须通过自定义输入描述故障** — 界面仅展示文本输入框，引导用户自行描述
3. **跨品类关键词匹配** — 系统将用户输入的关键词与全部已知品类的全部维修项目名称进行模糊匹配，尝试找到最接近的参考项目
4. **匹配成功**：使用匹配到的项目价格数据生成报告，并标注"该品类为近似匹配，仅供参考"
5. **匹配失败（无数据兜底）**：报告显示固定提示——"暂未收录该维修项目的市场价格，请谨慎参考，并建议咨询品牌官方售后。"AI 端收到 `noDataAvailable: true` 标记，仅生成通用建议而非价格分析

### 设计系统

## 10. 核心功能需求

### F1: 家电品类选择

- 展示 6 个品类：空调、冰箱、洗衣机、电视、热水器、其他
- 每个品类对应独立的维修项目列表和价格数据
- 选择后立即过渡到下一步

### F2: 型号输入（可选）

- 单行文本输入，Placeholder："例如：KFR-35GW（可选）"
- 明确标注"可跳过"，不阻断流程
- 型号信息用于未来价格数据细化，MVP 阶段仅做记录

### F3: 维修项目选择

- 根据所选品类动态加载预设维修项目
- 多选交互（标签形式）
- 支持自定义输入："其他故障，请描述…"
- 至少选择 1 项才能进入下一步

### F4: 报价输入

- 纯数字输入，支持小数点
- 单位：元（人民币）
- 校验：合理范围 1-100,000 元
- 一键进入分析

### F5: 报价分析报告

报告必须包含以下 12 个模块：

1. 本次维修内容摘要
2. 零件成本区间
3. 人工费用区间
4. 上门费用
5. 合理总价区间
6. 用户报价分析（合理/略高/偏高/严重偏高）
7. 溢价金额（如有）
8. 溢价比例（如有）
9. 市场参考区间
10. 风险提示（如：该品类故障可能涉及额外费用）
11. 下一步建议（如：建议咨询品牌官方售后比价、建议多问 1-2 家）
12. 免责声明

## 11. AI 能力设计

### 设计原则：结构化计算 + AI 解释

> **价格数据来自程序，解释文案来自 AI。绝不混淆。**

### AI 负责

- 理解用户自定义输入的维修项目，匹配到最近的已知项目
- 基于计算出的溢价数据，生成自然、有同理心的解释文案
- 生成个性化的"下一步建议"（根据溢价程度给出不同建议梯度）
- 生成风险提示（结合品类特性，提示可能的隐藏费用）
- 优化文案语气——做到"专业但不冰冷，友善但不油滑"

### AI 不负责（红线）

- ❌ 计算价格
- ❌ 推测价格
- ❌ 编造维修数据或市场行情
- ❌ 做绝对判断（如"你一定被骗了"）

### 为什么不设计成聊天机器人？

聊天界面在报价分析场景中存在三个致命问题：
1. **输入成本高** — 用户需要打字描述故障，不如点击选择高效
2. **不可预期** — 用户不知道要聊几轮才能得到结果，心理负担大
3. **可信度低** — 聊天机器人的回答看起来"像编的"，而结构化报告看起来"像分析的"

### 11.4 AI 价值论证：为什么不是模板？

> **核心问题：去掉 AI，这套系统还能用吗？**

能——但用户得到的是一份冷冰冰的数据报告，而非"决策支持"。以下用同一个场景对比：

**场景**：用户报告"空调不制冷，师傅说加氟+换电容要 1200 元"。计算引擎判定为"偏高"（市场参考 400-850 元）。

| 维度 | 模板方案（无 AI） | AI 方案（RepairLens） |
|------|------------------|----------------------|
| **结论呈现** | "您的报价偏高，溢价约 41%。" | "师傅报价 1200 元，比市场参考区间（400-850 元）高出约 41%。这个价格在品牌授权售后属于中上水平，但独立维修店通常可以更低。" |
| **风险提示** | 固定文本："维修可能存在额外费用。" | "加氟和换电容是空调常见故障，但需确认师傅是否排查了泄漏点——如果漏点未修复，加氟后很快会再次失效。建议要求师傅说明故障原因再决定维修方案。" |
| **下一步建议** | 固定文本："建议咨询多家维修店。" | "溢价约 350-800 元。建议：① 询问师傅是否包含高空作业费（如有，该报价可能合理）；② 致电品牌官方售后获取同项目报价对比；③ 如果急修且师傅提供保修，可尝试议价至 800-900 元。" |

**差距总结**：

- 模板能做到："告诉你贵了多少" → **信息告知**
- AI 能做到："解释为什么贵、贵在哪里、怎么判断值不值得修" → **决策支持**

这就是 RepairLens 需要 AI 的原因：不是告诉用户一个结论，而是帮助用户**理解**结论、**验证**结论、**行动**。

### 11.5 AI 输出质量标准

每份 AI 生成的报告文案需满足以下 5 项质量指标：

| 维度 | 标准 | 反例 |
|------|------|------|
| **事实准确** | AI 文案不得与计算引擎输出的结构化数据矛盾（如引擎判定溢价 41%，AI 不能说"略高一点"） | "价格还可以，不算贵"（实际溢价 50%） |
| **语气恰当** | 专业但不冰冷，友善但不油滑；站在消费者侧，不抹黑维修行业 | "你被宰了！赶紧拒绝！" |
| **建议可执行** | 每条建议包含具体动作和对象 | "可以再问问别人"（太模糊） |
| **字数控制** | 每个报告模块的 AI 文案 ≤ 150 汉字 | 长篇大论的分析段落 |
| **输出一致** | 相同输入 → 结构相似的输出；temperature 设为 0-0.3 | 同一场景两次分析给出矛盾结论 |

这些标准同时作为 Prompt Engineering 的约束条件嵌入 System Prompt。

### 价格数据模型 (`data/prices.json`)

```json
{
  "categories": {
    "air_conditioner": {
      "name": "空调",
      "repair_items": [
        {
          "id": "ac_compressor",
          "name": "压缩机更换",
          "parts_cost": [800, 2000],
          "labor_cost": [200, 500],
          "visit_fee": [30, 80],
          "notes": "变频压缩机价格偏高，定频偏低"
        }
      ]
    }
  }
}
```

### 价格数据来源与初始化策略

**MVP 数据采集方法**（三种来源交叉验证）：

1. **品牌官方售后价目表**：公开可查的各品牌授权服务商报价（上界参考）
2. **电商平台零件价格**：京东/天猫官方旗舰店的正品配件零售价（零件成本基准）
3. **社区经验数据**：小红书、知乎等平台用户分享的实际维修费用（市场真实成交参考）

**初始化范围**：6 品类 × 5-10 个维修项目，约 40-50 条数据记录，每条包含三档价格区间。

**数据维护**：
- 版本化管理，每次修订标注 `last_updated` 日期
- 季度人工审核更新（MVP 阶段）
- 未来通过用户反馈数据自动校准

**设计考量：**
- 价格使用**区间**而非固定值，体现市场真实波动
- 每个维修项目包含零件成本、人工成本、上门费三项独立字段
- `notes` 字段为 AI 解释提供领域知识上下文
- JSON 格式易于维护和扩展，非技术人员也可编辑

### 多选维修项目的价格合并规则

当用户选择多个维修项目时，价格计算遵循以下规则：

| 费用类型 | 合并规则 | 原因 |
|---------|---------|------|
| 零件成本 | **各项叠加** — `[Σmin, Σmax]` | 每个项目的零件都需要单独购买 |
| 人工费用 | **各项叠加** — `[Σmin, Σmax]` | 不同维修项目可能需要不同的工序和工时 |
| 上门费用 | **取最大值** — `[max(min), max(max)]` | 一次上门服务覆盖所有维修，技师收取一项最高上门费 |
| 总价区间 | `[零件Σmin+人工Σmin+上门max, 零件Σmax+人工Σmax+上门max]` | 合并以上三项 |

**特殊处理**：
- 若某项目为自定义项目（无匹配价格数据），从合并计算中排除，但在报告中标记为"未知项目"
- 若全部项目均为自定义/未知，报告进入无数据兜底模式，显示："暂未收录该维修项目的市场价格，请谨慎参考，并建议咨询品牌官方售后。"

## 13. 技术方案概览

| 层级 | 技术选型 | 选型理由 |
|------|---------|---------|
| 前端 | React (Vite) + Tailwind CSS | 轻量、移动端适配友好、生态成熟 |
| AI | Claude API (结构化 Prompt) | 通过 System Prompt 严格约束 AI 职责边界 |
| 价格计算 | 纯前端/Edge Function 规则引擎 | 确定性计算，不依赖 AI |
| 价格数据 | 本地 JSON 文件 | MVP 简单可控，未来可迁移到数据库 |
| 部署 | Vercel / Cloudflare Pages | 免费额度充足，全球 CDN |

### 架构约束

- 价格计算与 AI 解释**物理隔离**——先计算、后解释，AI 接收的是已计算好的数据
- AI Prompt 中明确禁止编造价格，仅基于传入的结构化数据进行解释
- 所有价格数据的最终来源均为 `data/prices.json`，可审计、可追溯

## 15. 风险与边界

### 产品边界

- **不做绝对判断。** RepairLens 提供价格分析和建议，但最终决策权始终在用户手中。分析结论使用"合理/略高/偏高/严重偏高"四档而非"对/错"二值判断。
- **不替代维修人员。** RepairLens 不诊断故障原因，不推荐维修方案，只分析报价。
- **价格数据仅供参考。** 实际价格因地区、品牌、季节、配件来源等因素而异。免责声明必须显著展示。

### 关键风险

| 风险 | 应对策略 |
|------|---------|
| 价格数据不准确引发用户投诉 | 使用区间而非固定值；显著展示免责声明；持续更新数据 |
| AI 幻觉编造价格 | 架构上隔离计算与解释；Prompt 红线约束；价格字段硬编码 |
| 维修师傅/OEM 投诉 | 明确声明"仅供参考，不构成官方报价"；不贬低维修行业 |
| 用户过度依赖工具判断 | 报告中始终包含"建议多方比价"的提示 |

### 道德底线

1. **永远站在消费者一侧，但不抹黑维修行业。** 维修师傅的合理利润应该被尊重，RepairLens 只帮助识别明显不合理的溢价。
2. **数据透明。** 所有价格区间的来源逻辑应对用户可见（如"基于市场公开数据"）。
3. **不做恐慌营销。** 不用"你被宰了！"等煽动性语言，即使报价严重偏高，也以事实陈述为主。

> **RepairLens 的终极目标：不是让用户省最多的钱，而是让用户每一次维修消费都明明白白。**
